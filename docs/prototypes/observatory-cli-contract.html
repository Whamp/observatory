Workflow verbs · superseded

Start with what the agent wants to do, then name the resource. This reads like imperative prose: obs publish create, obs register service, obs remove artifact.

TradeoffExcellent for the first publish. Weak for discovery: an agent must search several verb families to learn everything it can do to one resource.

B

Resource namespaces · approved

Choose a noun, then see every legal operation on it: obs artifact publish, obs service register, obs service target add. The hierarchy mirrors the domain language and keeps strict create/update intent visible.

ApprovedThis is the smallest self-documenting interface for agents. Six top-level nouns expose the whole model; each noun’s help is a complete capability inventory. No generic CRUD hides Publish, teardown, restore, refresh, or keep.

USAGE  obs [GLOBAL OPTIONS] <RESOURCE> <ACTION>

GLOBAL OPTIONS
  --server URL            default http://127.0.0.1:3773

EXAMPLES

CONTRACT
  Create is strict. Updates and replacements are explicit.
  Returned URLs are canonical; never construct one from an ID.

  Non-TTY invocations never prompt.

C

Intent commands · superseded

Put lifecycle intent at the root: obs create artifact, obs replace artifact, obs update service. Reads use get; operational actions use act. The grammar is compact and maps cleanly to request objects.

TradeoffBest for generated clients and request manifests. Worst at the terminal: domain-specific operations hide behind act, and get reduces the visibility of Artifact versus Service capabilities.
Grammar
obs <intent> <kind> [identity] [fields]. The help system is organized around mutation semantics rather than resources.

Verdict

C makes API intent obvious and keeps create versus update strict. But act becomes a miscellaneous drawer containing nine unrelated verbs. Agents asking “what can I do with a Service?” must cross create, update, get, delete, and act. The root is compact; discovery is not.
